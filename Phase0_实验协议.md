# Phase 0 实验协议

**暂定题目**：《面向约束旅游规划的目标驱动多智能体协同调度方法》  
**协议版本**：CTP-GMAS-EXP-v1.2  
**日期**：2026-07-16  
**协议性质**：本项目后续代码、数据、实验和论文共同遵守的总规则  
**现实约束**：单人、两周、LLM API 总预算不超过 150 元

## 一、项目共同宗旨

本项目不是继续展示一个“能够生成旅游攻略”的毕设系统，而是形成一项可以被客观验证的多 Agent 学术成果。

后续工作必须遵守以下原则：

1. **只研究一个核心问题**：多 Agent 如何根据用户目标和状态变化动态选择 Agent 与工具。
2. **所有结论必须有实验数据支持**，不能依靠案例展示或系统自身 Review 分数。
3. **四种方法必须公平比较**，使用相同模型、数据、工具、输出要求和资源限制。
4. **正式实验必须可复现**，不依赖不稳定的实时旅游 API。
5. **测试集冻结后不得调参**，正式结果不得手工修改或选择性删除。
6. **不夸大结论**，实验没有证明的能力不得写入论文贡献。

## 二、五个问题的唯一答案

| 必须回答的问题 | 本项目的唯一答案 |
|---|---|
| 论文提出什么方法？ | 提出“目标—状态驱动的自适应多 Agent 协同调度方法”：统一识别任务和状态变化，根据能力依赖选择最小必要 Agent 与工具，并复用未失效结果。 |
| 四种方法有什么区别？ | M0 无 Agent、无工具；M1 一个通用 Agent 自主用工具；M2 固定执行全部业务 Agent；M3 根据目标和状态动态选择 Agent、工具并复用结果。 |
| 什么叫一个案例成功？ | 方法输出结构合法、任务处理正确、适用硬约束全部满足、必要证据真实且必要工具成功；不同任务只检查其适用条件。 |
| 记录哪些 Agent 性能指标？ | Agent 选择 F1、完全匹配率、必要 Agent 覆盖率、多余/重复 Agent 数、计划与实际执行一致率、调用成功率、调用次数、耗时、Token 和费用。 |
| 使用多少案例？ | 共 120 个：20 个开发案例用于调试，100 个冻结测试案例用于正式统计。 |

以上五项是项目的固定研究口径。任何修改都必须先更新协议版本并说明原因。

## 三、论文提出的方法

论文方法统一命名为：

> **目标—状态驱动的自适应多 Agent 协同调度方法**

该方法包含三个核心步骤：

1. **统一目标决策**  
   把当前对话转换为结构化任务单，包括任务类型、已知条件、缺失条件、本轮变化条件、应保留条件、所需能力和证据。

2. **依赖驱动调度**  
   根据任务单，从景点、天气、行程、预算 Agent 中选择完成任务所需的最小子集，并确定需要调用的离线工具。

3. **状态感知复用**  
   多轮对话中，只重新执行受到条件变化影响的 Agent；输入和上游结果没有失效时，复用已有结果。

论文研究三个问题：

- **RQ1 调度正确性**：任务、状态变化、Agent 和工具选择是否正确？
- **RQ2 任务正确性**：结果是否满足硬约束并具有离线数据证据？
- **RQ3 执行效率**：相比固定多 Agent 链路，是否减少调用、Token、时延和费用？

论文不评价语言是否优美，也不使用系统自身 Review 分数作为主要证据。

## 四、四种实验方法

| 维度 | M0 Direct LLM | M1 Single Agent | M2 Fixed Multi-Agent | M3 Proposed Method |
|---|---|---|---|---|
| Agent 结构 | 无 Agent | 1 个通用 Agent | 4 个业务 Agent | 4 个业务 Agent |
| 工具使用 | 不使用 | 自主选择离线工具 | 各 Agent 使用对应工具 | 根据目标动态选择工具 |
| 调度方式 | 无 | 单 Agent 内部决策 | 固定完整链路 | 动态最小必要子集 |
| 多轮状态变化 | 只读取输入上下文 | 由单 Agent 处理 | 原则上重新执行完整链路 | 识别变化并局部重算 |
| 旧结果复用 | 无 | 无 | 无 | 复用未失效结果 |
| 实验作用 | 无工具能力下限 | 单 Agent 基线 | 固定多 Agent 基线 | 论文提出的方法 |

首要比较是 **M3 与 M2**。两者使用相同业务 Agent、工具和模型，核心区别只能是“固定执行”与“目标—状态驱动调度”。M0、M1 作为辅助基线。

公平规则：

- 使用相同模型版本、温度、Token 上限、超时和重试规则；
- 使用相同离线数据、工具实现、输出 Schema 和评价器；
- 各方法可以使用适合自身结构的 Prompt，但必须在正式实验前冻结并记录版本；
- 每个案例内的方法运行顺序按固定随机种子打乱；
- 禁止 Mock、静默 fallback 和未记录的缓存结果。

## 五、案例与数据

数据集名称暂定为 **CTP-120**：

- **20 个开发案例**：仅用于代码、Prompt 和费用调试，不进入正式结果；
- **100 个测试案例**：用于正式统计，冻结后禁止调参和试跑。

案例覆盖六类任务：

1. 完整旅游规划；
2. 信息不完整与澄清；
3. 景点、天气或预算单项任务；
4. 多轮条件修改；
5. 天气变化与行程调整；
6. 闲聊或无关问题。

使用 4～5 个数据完整的城市。每个案例必须标注：

- 任务类型及对话上下文；
- 已知、缺失、变化和应保留的条件；
- 所需能力与证据；
- 一个或多个可接受的 Agent 集合；
- 必需和禁止使用的工具；
- 可由程序检查的硬约束；
- 对应离线数据版本。

正式实验只读取固定的 POI、天气和预算数据，不访问实时高德、天气或其他旅游 API。

案例由模板辅助生成，再由作者分两轮核查。测试集冻结前必须排除：

- 重复或高度近似案例；
- 无法由离线数据支持的事实；
- 无法自动评价的约束；
- 开发集与测试集泄漏。

## 六、一个案例怎样才算成功

首要指标为 **严格任务成功率（STSR）**。每个案例只有满足该任务适用的全部条件才记为成功。

| 任务类型 | 成功条件 |
|---|---|
| 完整规划 | 输出结构合法、任务正确、全部硬约束通过、POI 有离线证据、必要工具成功 |
| 信息不完整 | 正确指出缺失信息并发起澄清，不提前生成虚假的完整计划 |
| 单项任务 | 正确完成指定任务，并且不执行无关的完整规划链 |
| 多轮修改 | 正确更新变化条件、保留未变化条件，最终结果满足全部适用约束 |
| 天气调整 | 使用对应天气证据，正确调整受影响内容并保留其他有效结果 |
| 闲聊/无关问题 | 正确回复且不调用旅游业务 Agent 和工具 |

不适用的条件记为 `NA`，不进入成功判断或约束分母。

次要正确性指标：

- **硬约束满足率 HCSR**：每个案例通过的适用硬约束数除以适用硬约束总数，再对案例等权平均；
- POI 证据支持率；
- 虚构 POI 比例；
- 变化条件识别 F1；
- 输出结构合法率。

论文评价器必须独立运行：

- 只在方法输出完成后评分；
- 不参与生成、重试或修复；
- 不把违规信息返回给任何方法；
- 不读取方法名称；
- 相同输入始终得到相同评分。

## 七、必须记录的 Agent 与工具指标

### 1. Agent 调度指标

- Agent 选择 Precision、Recall、F1；
- Agent 集合完全匹配率；
- 必要 Agent 执行覆盖率；
- 多余 Agent 调用数；
- 重复 Agent 调用数；
- 计划 Agent 与实际 Agent 的一致率；
- Agent 执行成功率；
- 各 Agent 调用次数和执行耗时。

存在多个可接受 Agent 集合时，取与任一合法集合匹配的最高分。Agent 指标使用实际执行集合；重复调用不进入集合 F1，而单独计数。

### 2. 工具调用指标

- 工具选择 Precision、Recall、F1；
- 工具集合完全匹配率；
- 必要工具执行覆盖率；
- 多余、重复和禁止工具调用数；
- 计划工具与实际工具的一致率；
- 工具调用成功率及失败类型。

### 3. 系统开销指标

- LLM、Agent 和工具调用次数；
- 输入 Token、输出 Token 和总 Token；
- 端到端时延；
- 各阶段耗时；
- 单案例费用；
- 每个成功案例的平均费用。

## 八、正式实验规则

基础主实验为：

```text
100 个测试案例 × 4 种方法 × 1 次 = 400 次运行
```

预算允许时，M2、M3 优先在全部测试案例重复 3 次；否则在预先选定的 24 个分层案例上重复 3 次。重复结果先按案例聚合，不能当作额外独立样本。

统计至少报告：

- 配对差值；
- 95% bootstrap 置信区间；
- 中位数和四分位距；
- 二值指标的 McNemar 检验；
- 连续指标的 Wilcoxon 配对检验。

样本量不足时只能报告“正确性保持情况”，不得声称统计学非劣。

运行纪律：

- 模型逻辑错误、格式错误和幻觉计为失败，不得重新运行到成功；
- 仅超时、HTTP 429 或 5xx 最多重试 2 次；
- 重试产生的 Token、时延和费用全部计入；
- 原始输出、Trace 和实验 manifest 只增不改；
- 汇总结果必须由程序从原始数据生成，禁止手工修改。

## 十、结论边界

如果 M3 保持正确性并显著降低调用或成本，可以得出“动态调度提高执行效率”的结论。

如果 M3 只降低成本但正确性没有提高，只能讨论“质量—成本权衡”；如果正确性明显下降，不得声称在保持质量的同时提高效率。

本文不研究或声称：

- RAG 或长期记忆；
- 真实路线全局最优；
- 酒店、机票等真实完整价格；
- 真实用户偏好或人工满意度优势；
- 全国城市普适性；
- 实时旅游数据准确性；
- 商业部署可用性。

3. 每类案例的严格成功条件已经明确；
4. Agent、工具、正确性和系统开销指标已经明确；

后续代码实现、数据构造、实验运行、结果分析和论文写作均以本协议为共同宗旨。若必须修改核心规则，应升级协议版本并记录修改原因，不能在正式实验后根据结果临时改变评价标准。

## 十二、第 3 天统一工具与输出补充

本项目正式实验固定使用三个生成工具和一个独立评价工具。

生成阶段工具：

| 工具名 | 作用 | 使用方法 |
|---|---|---|
| `poi_search` | 根据城市、偏好和人群查询固定离线景点 | M1、M2、M3 可调用 |
| `weather_query` | 根据城市和日期查询固定离线天气 | M1、M2、M3 可调用 |
| `budget_calculator` | 根据人数、天数、景点和消费等级计算结构化预算 | M1、M2、M3 可调用 |

独立评价工具：

| 工具名 | 作用 | 使用方法 |
|---|---|---|
| `constraint_checker` | 检查天数、预算、天气调整、景点数量等硬约束 | 只在方法生成完成后由评价器调用，不暴露给生成方法 |

工具返回统一使用 `research_tool_result_v1`。方法输出统一封装为 `ctp-experiment-output-v1`，必须包含任务类型、使用的 Agent、调用的工具、旅行天数、每日行程、预算、天气调整、约束报告和执行状态。

工具状态定义：

| 状态 | 含义 | 是否计为工具调用成功 |
|---|---|---|
| `success` | 输入合法且固定数据命中 | 是 |
| `no_result` | 输入合法但固定数据无匹配项 | 是，但任务评价可能失败 |
| `failed` | 参数错误、城市不支持、日期格式错误或固定数据缺失 | 否 |

工具计数规则：

- 参数错误也必须计入一次失败工具调用；
- 缓存命中仍计为工具调用，但外部 API 调用数必须为 0；
- M0 `llm_direct` 的工具调用次数必须严格为 0；
- `constraint_checker` 不计入生成阶段 Tool F1。

固定数据原则：

- 天气必须由“城市 + 日期”唯一决定；若传入 `scenario_type` 但同时传入真实日期，以城市和日期的固定映射为准；
- 预算中未知成人门票价格不能按 0 元处理，必须使用明确标注的实验估算值，并在 `ticket_breakdown` 中记录估算规则；
- 每次实验结果必须记录输入哈希、结果哈希和固定数据文件哈希摘要，用于证明可复现性。
